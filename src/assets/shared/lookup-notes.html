Compilation, caching, and freshness

This applies to all four lookup functions: VLOOKUP and VLOOKUP_CONNECTION (the VLOOKUP family), and VLOOKUP_RECORD and VLOOKUP_RECORD_CONNECTION (the VLOOKUP_RECORD family).

Both families compile and merge invocations into consolidated SOQL queries — calls that share the same object, WHERE, and additional_criteria (differing only in the SELECT fields) are combined to minimize database calls. The difference is in scope and caching:

VLOOKUP family — compiles and merges at the Executable level only. Results do not cache beyond the current Executable's execution. Each Executable issues its own fresh query, returning the most recent committed data.
VLOOKUP_RECORD family — compiles and merges across all Executables that belong to the same triggering object. The query is issued once and cached afterwards; subsequent invocations return the cached result without issuing separate queries. If a prior Executable has updated the looked-up records, the cached value will be stale.

When to use which. Use the VLOOKUP family to issue fresh queries and get the most recent data — for example, when prior Executables on the same triggering object may have updated the target records. Use the VLOOKUP_RECORD family when minimizing SOQL usage matters and the cached value is acceptable.

ORDER BY behavior

DSP automatically appends an ORDER BY clause to the SOQL generated by all four lookup functions (VLOOKUP, VLOOKUP_CONNECTION, VLOOKUP_RECORD, VLOOKUP_RECORD_CONNECTION), based on the selected fields, to improve lookup performance.

If you include your own ORDER BY in additional_criteria, DSP uses your clause as written and does not add its own. This lets you override the default ordering whenever needed.
